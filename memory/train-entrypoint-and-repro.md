**Entry point:** `python train.py --config ml/configs/{addis|porto_seguro}.yaml [--model rf|xgb|baseline|all]` is the single reproducible pipeline (load→clean→engineer→split→train w/ SMOTE-in-CV→evaluate→SHAP→summary). It's a thin wrapper over `ml/`; each run writes a self-contained dir `ml/artifacts/runs/{timestamp}_{git-sha}/` (processed data, model+meta, encoders, feature_contract, reports/, resolved_config.json, config.yaml, run.log, summary.json). The `runs/` dirs are under the gitignored `ml/artifacts/`.

**Reproducibility gotcha (non-obvious):** RandomForest with `n_jobs=-1` sums per-tree probabilities across threads in nondeterministic order, perturbing `predict_proba` (hence ROC-AUC) at ~1e-7 — enough to break bit-exact metric reproducibility while leaving argmax predictions (F1/recall/accuracy) identical. Fix: RF runs with `n_jobs=1` ([ml/training/train_rf.py]). XGBoost (`tree_method=hist`) and the logistic/rule baselines were already deterministic. Both datasets now reproduce bit-identical metrics across runs with the same config+seed.

**Binary support is config-only:** the framework is `kind`-aware (config `n_classes`, binary ROC-AUC, kind-aware XGBoost objective, `baseline.kind` = rule_based|logistic). Adding a dataset = a new YAML, no code changes. **Porto data is SYNTHETIC** — `data/raw/porto_seguro_sample.csv` is generated by `scripts/make_porto_sample.py` (real Porto schema, ~12% positive) because the real Kaggle set isn't downloaded. Drop the real `train.csv` at that path to switch — config lists a subset of real columns; extras are ignored. See [[phase1b-modelling-results]], [[data-git-hygiene]].
